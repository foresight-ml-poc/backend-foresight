## 1. Contexte

[Foresight](https://github.com/vcapton-jpg/polymarket-ai) (alias polymarket-ai)
est une plateforme d'intelligence temps réel sur Polymarket : elle ingère des
news, les regroupe en événements, fait du hybrid search sur les marchés
correspondants, fait analyser l'impact par GPT-4o, et émet un **signal** noté
0-100 indiquant à un trader s'il faut acheter YES ou NO.

Aujourd'hui, le score est calculé par une **formule heuristique** :

```
signal_strength = 0.15·freshness + 0.10·source_weight + 0.15·confirmation
                + 0.60·(0.65·impact_strength + 0.35·llm_confidence)
trade_quality   = 0.40·liquidity + 0.35·spread + 0.25·time_to_resolution
signal_score    = 0.75·signal_strength + 0.25·trade_quality
```

L'audit interne Foresight (BLUEPRINT.md, section 17.2) montre un **winrate
global à T+24h de 46-48 %** — sous le hasard. Le scoring heuristique a atteint
sa limite : il ne capture pas les non-linéarités ni les interactions entre
features.

**Ce projet remplace cette formule par du machine learning supervisé.**

---

## 2. Problème ML

**Tâche :** classification binaire supervisée.

**Cible (label) :** `signal_outcomes.direction_correct` — pour chaque signal
historique, le mouvement de prix réel à T+24h confirmait-il la direction
prédite ? Boolean → 0 / 1.

**Pourquoi ce label plutôt qu'un autre :**

- Disponible **24h après l'émission** du signal → beaucoup plus de samples
  que `outcome_label` (résolution finale du marché, qui peut prendre des
  semaines).
- Naturellement aligné avec ce qu'on veut remplacer : "est-ce que ce signal
  aurait été gagnant ?"
- Compatible avec les 3 types de modèles (LogReg / RF / MLP).

**Volume de données attendu :** entre 200 et 2000 signaux historiques avec
`direction_correct IS NOT NULL`. Sweet spot pour comparer 3 modèles sans
synthétique.

## 3. Décisions prises pendant le brainstorming

Récap des 5 questions structurantes (référence : conversation 2026-05-05) :

| # | Question | Choix | Justification |
|---|---|---|---|
| 1 | Nature du projet | POC pédagogique + intention de réutiliser plus tard | Le best model pourra revenir dans Foresight pour remplacer (ou compléter) la formule actuelle |
| 2 | Cible de prédiction | `direction_correct` à T+24h | Maximum de samples disponibles, label binaire propre, narrative claire pour le jury |
| 3 | Volume de données | 200-2000 signaux | Permet train/val/test classique sans synthétique |
| 4 | Structure des repos | 3 repos séparés (ml + backend + frontend), repo ML calqué 1:1 sur Basile | Respecte la consigne école "split en 3 parties" et garde le repo ML aligné sur la référence |
| 5 | 3e modèle | MLP simple (Keras) | Touche au DL sans sur-promettre, reste comparable en tâche binaire |

**Métriques retenues :** accuracy, precision, recall, F1-score, ROC-AUC,
matrice de confusion, log-loss. Critère de sélection du best model :
**ROC-AUC** sur le validation set.

**Comparaison vs baseline :** l'heuristique actuelle (formule de Foresight) est
recalculée sur le test set comme **baseline à battre**. Si le ML ne la bat pas,
on l'écrit honnêtement dans le rapport.

### 6.2 `src/data.py::load_dataset_split()` — implémentation

Signature **strictement** celle de Basile. À l'intérieur, on fait notre
cleaning + feature engineering + split :

```python
from __future__ import annotations
from typing import Any
import pandas as pd
from sklearn.model_selection import train_test_split
from sklearn.preprocessing import StandardScaler
import joblib

from config import DATA_DIR, MODELS_DIR, SEED, TARGET_COLUMN


def load_dataset_split() -> tuple[Any, Any, Any, Any]:
    """Charge le CSV exporté de Foresight, applique cleaning + feature engineering,
    retourne (X_train, X_test, y_train, y_test).

    Le scaler est fit sur X_train uniquement et sauvegardé en models/scaler.pkl
    pour que le backend puisse appliquer la même transformation à l'inférence.
    """
    # 1. Load — privilégie le dernier export, fallback sur le sample committé
    raw_csvs = sorted((DATA_DIR / "raw").glob("signals_export_*.csv"))
    csv_path = raw_csvs[-1] if raw_csvs else (DATA_DIR / "raw" / "signals_export_sample.csv")
    df = pd.read_csv(csv_path)

    # 2. Clean — drop NULL labels, drop NULL features critiques, cast types
    df = _clean(df)

    # 3. Feature engineering — dérivées + one-hot bucket + sélection
    X, y = _feature_engineer(df)

    # 4. Split stratifié 70/15/15 → (train+val) / test puis val depuis train
    X_train, X_test, y_train, y_test = train_test_split(
        X, y, test_size=0.15, stratify=y, random_state=SEED
    )

    # 5. Scaling — fit sur train uniquement, save scaler.pkl
    scaler = StandardScaler()
    X_train_scaled = scaler.fit_transform(X_train)
    X_test_scaled = scaler.transform(X_test)
    joblib.dump(scaler, MODELS_DIR / "scaler.pkl")
    joblib.dump(list(X.columns), MODELS_DIR / "feature_order.pkl")

    return X_train_scaled, X_test_scaled, y_train.values, y_test.values


def _clean(df: pd.DataFrame) -> pd.DataFrame:
    """Drop invalid rows, cast types."""
    # Drop rows where label is NULL
    df = df.dropna(subset=[TARGET_COLUMN])
    # Drop rows where critical features are NULL

    return df
```

### 6.3 `src/metrics.py::compute_metrics()` — implémentation

Signature **strictement** celle de Basile. À l'intérieur, on calcule nos
métriques de classification binaire :

```python
from __future__ import annotations
from typing import Any
from sklearn.metrics import (
    accuracy_score, precision_score, recall_score,
    f1_score, roc_auc_score, log_loss
)


def compute_metrics(y_true: Any, y_pred: Any) -> dict[str, float]:
    """Métriques de classification binaire pour direction_correct ∈ {0, 1}.

    Note : si le modèle expose predict_proba, scripts/main.py utilise predict.
    Donc y_pred est binaire ici. ROC-AUC sur predictions binaires est moins
    précis que sur les probas — mais reste un proxy utile pour la comparaison.
    Pour le ROC-AUC sur probas, voir notebooks/04_results_analysis.ipynb.
    """
    return {
        "accuracy": accuracy_score(y_true, y_pred),
        "precision": precision_score(y_true, y_pred, zero_division=0),
        "recall": recall_score(y_true, y_pred, zero_division=0),
        "f1": f1_score(y_true, y_pred, zero_division=0),
        "roc_auc": roc_auc_score(y_true, y_pred),
    }
```

## 7. `scripts/train.py` — le script qui entraîne les 3 modèles

### 7.1 Pourquoi un script séparé de `scripts/main.py`

`scripts/main.py` (fixé par Basile) sert UNIQUEMENT à **évaluer** des modèles
déjà entraînés et à lancer Streamlit. Il faut donc un script séparé pour
**entraîner** les modèles et générer les `.pkl` que Basile va charger.

### 7.2 Pipeline de `scripts/train.py`

```
1. load_dataset_split() depuis src/data.py
   → (X_train, X_test, y_train, y_test) déjà scaled
2. Pour chaque famille [LogReg, RandomForest, MLP] :
     a. Tuning hyperparamètres via 5-fold CV sur X_train
     b. Refit sur X_train complet avec best_params
     c. Évaluation sur X_test (juste pour log, pas de leakage)
     d. joblib.dump(model, MODELS_DIR / "<name>.pkl")
3. Sélection best model via val_roc_auc
   → copie en models/best_model.pkl
4. Génération models/model_card.json :
     - feature_order, feature_dtypes, scaler_path
     - test_metrics du best model
     - heuristic_baseline (recalculé avec src/compare ou inline)
5. Génération plots/ :
     - confusion_matrix_<name>.png pour chaque modèle
     - roc_curves_comparison.png (3 courbes + heuristique)
     - feature_importance_rf.png
     - learning_curves.png (au moins pour MLP)
     - ml_vs_heuristic.png (la punchline)
```

**Modèle 3 — MLP (Keras wrappé)**
```python
def build_mlp(n_features: int, learning_rate: float, dropout_a: float, dropout_b: float):
    model = Sequential([
        Input(shape=(n_features,)),
        Dense(64, activation='relu'),
        BatchNormalization(),
        Dropout(dropout_a),
        Dense(32, activation='relu'),
        Dropout(dropout_b),
        Dense(1, activation='sigmoid'),
    ])
    model.compile(
        optimizer=Adam(learning_rate=learning_rate),
        loss='binary_crossentropy',
        metrics=['accuracy', tf.keras.metrics.AUC(name='auc')],
    )
    return model

# Wrapper pour exposer .predict(X) → array binaire (compatible Basile main.py)
class KerasMLPWrapper:
    def __init__(self, model: tf.keras.Model, threshold: float = 0.5):
        self.model = model
        self.threshold = threshold

    def predict(self, X) -> np.ndarray:
        proba = self.model.predict(X, verbose=0).ravel()
        return (proba >= self.threshold).astype(int)

    def predict_proba(self, X) -> np.ndarray:
        proba = self.model.predict(X, verbose=0).ravel()
        return np.column_stack([1 - proba, proba])
```

Le wrapper est sérialisé via `joblib.dump(KerasMLPWrapper(model), "mlp.pkl")`.
Si joblib ne sait pas sérialiser le tf.keras.Model directement, on sauve
le `.h5` séparément et on charge dans `__init__` du wrapper.

### 7.4 Garde-fous anti-leak

- **Pas de feature dérivée du futur** : aucune feature ne touche à
  `move_t24h_pct`, `price_t24h`, `outcome_label`, `price_resolved` (cf.
  `EXCLUDED_FROM_FEATURES` dans `src/config.py`)
- **Pas de feature dérivée du score actuel** : on n'utilise PAS `signal_score`,
  `signal_strength`, `trade_quality` comme feature (idem)
- **Scaler fit sur train uniquement**
- **5-fold CV sur le train set pour le tuning, jamais sur le test**

## 8. Données : extraction, cleaning, feature engineering

### 8.1 Tables sources dans Foresight

D'après `BLUEPRINT.md` section 5 :

| Table | Colonnes utilisées |
|---|---|
| `signals` | id, event_id, market_id, direction, market_price_at_signal, source_tier_mix (JSONB), created_at |
| `signal_outcomes` | direction_correct (**LE LABEL**), move_t24h_pct (analyse seulement) |
| `event_market_features` | freshness, source_weight, confirmation, liquidity, spread, time_to_resolution |
| `event_market_analysis` | impact_strength, confidence, ambiguity_score, specificity_score |
| `events` | bucket, articles_count, unique_sources_count |

### 8.2 Requête d'export (`scripts/export_from_foresight.py`)

```sql
SELECT
  s.id AS signal_id,
  s.created_at,
  s.direction,
  s.market_price_at_signal,
  s.source_tier_mix,
  emf.freshness, emf.source_weight, emf.confirmation,
  emf.liquidity, emf.spread, emf.time_to_resolution,
  ema.impact_strength, ema.confidence AS llm_confidence,
  ema.ambiguity_score, ema.specificity_score,
  e.bucket, e.articles_count, e.unique_sources_count,
  so.direction_correct,
  so.move_t24h_pct
FROM signals s
JOIN signal_outcomes so ON so.signal_id = s.id
JOIN event_market_features emf
  ON emf.event_id = s.event_id AND emf.market_id = s.market_id
JOIN event_market_analysis ema
  ON ema.event_id = s.event_id AND ema.market_id = s.market_id
JOIN events e ON e.id = s.event_id
WHERE so.direction_correct IS NOT NULL
  AND s.below_threshold = FALSE
  AND emf.freshness IS NOT NULL
ORDER BY s.created_at;
```

Sortie : `data/raw/signals_export_2026-MM-DD.csv` (~20 colonnes, 200-2000 lignes).

**Sample anonymisé commité** : `data/raw/signals_export_sample.csv` (50 lignes)
pour qu'un évaluateur puisse rejouer le pipeline sans accès DB.

### 8.3 Cleaning (`src/data.py::_clean()`)

1. **Drop des lignes invalides** : `direction_correct` NULL, features critiques
   NULL
2. **Cast types** : `direction_correct` → int (0/1), JSONB `source_tier_mix`
   → 3 colonnes (tier_1_count, tier_2_count, tier_3_count)
3. **Outliers** : capping à p99 sur `move_t24h_pct` (analyse uniquement, pas
   en feature)
4. **Sanity checks** :
   - features [0,1] sauf counts
   - `direction_correct` ∈ {0, 1}
   - ≥2 classes représentées (sinon erreur explicite)

### 8.4 Feature engineering (`src/data.py::_feature_engineer()`)

**Features brutes (12) — déjà dans le CSV :**
- 6 heuristiques : freshness, source_weight, confirmation, liquidity, spread, time_to_resolution
- 4 LLM : impact_strength, llm_confidence, ambiguity_score, specificity_score
- 2 contexte event : articles_count, unique_sources_count

**Features dérivées qu'on construit (5) :**
- `tier_1_count`, `tier_2_count`, `tier_3_count` — depuis JSONB `source_tier_mix`
- `is_buy_yes` — `direction == "BUY_YES"` (capture l'asymétrie BUY_NO 72.7 %
  vs BUY_YES 29.2 % observée dans l'audit Foresight)
- `hour_of_day` — extrait de `created_at`, capture les patterns horaires
- `market_price_centered` — `|market_price_at_signal - 0.5|`, mesure
  l'incertitude initiale du marché

**Features catégorielles (1, one-hot) :**
- `bucket` : geopolitics / politics / sports / crypto / other → 4-6 colonnes
  binaires (`pd.get_dummies(..., drop_first=True)`)

**Total final ≈ 18-22 features.**

## 11. Workflow et livrables

### 11.1 Ordre de construction (4 phases)

**Phase 1 — Setup data + ml-foresight repo (semaine 1-2)**

7. Implémenter `src/data.py::load_dataset_split()` (cleaning + feature eng + split)

14. `backend-foresight` : FastAPI + model_loader + /predict
15. Tests pytest qui passent
16. Dockerfile + `docker run` testé en local

**Phase 4 — Frontend + démo (semaine 4, ~3 jours)**

18. `frontend-foresight` : init Vite + 2 vues
19. Charge sample CSV, appelle backend, affiche comparaison
20. Streamlit (dans ml repo) finalisé pour la soutenance

## 12. Risques identifiés et mitigations

| Risque | Probabilité | Impact | Mitigation |
|---|---|---|---|
| Dataset trop petit (<200 signaux exploitables) | Moyenne | Élevé | Élargir la query, oversampling SMOTE |
| Déséquilibre des classes (>70/30) | Moyenne | Moyen | `class_weight='balanced'`, F1+ROC-AUC |
| Le ML ne bat pas l'heuristique | Faible-moyenne | Moyen | Écrit honnêtement dans le rapport |
| Stationnarité (patterns du passé ne tiennent plus) | Élevée | Faible (POC) | Mentionné dans la discussion |
| Leak de feature | Moyenne | Critique | `EXCLUDED_FROM_FEATURES` dans config.py + code review |
| Wrapper Keras incompatible avec joblib | Moyenne | Moyen | Sauver `.h5` séparément + reconstruction dans wrapper.__init__ |
| Modèle trop gros pour GitHub Release (>2GB) | Faible | Moyen | Quantization ou Git LFS |
